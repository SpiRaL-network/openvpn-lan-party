# Security policy

## Reporting a vulnerability

Please report security issues privately through GitHub's **Security > Report a
vulnerability** feature when it is available. Do not open a public issue that
contains credentials, private keys, VPN profiles, public IP addresses, bearer
tokens or TPM/CNG diagnostic identifiers.

## Files that must never be committed

Never publish files generated by a deployed server, especially:

- `/root/openvpn-pki/`;
- `/var/lib/openvpn-lan-party/enrollment/`;
- `/var/lib/openvpn/credential-registry.json` from a real deployment;
- every generated `companion.json` and its bearer token;
- `/etc/openvpn-lan-companion/` deployment state;
- client certificates, private keys, `.ovpn` profiles or enrolment responses.

The repository contains templates only. Every deployment must generate a new
certificate authority and unique player identities.

## Trusted-member boundary

Issue VPN profiles only to people the administrator knows and trusts. All
connected players share one Layer-2 virtual LAN; Companion permissions protect
application identity and lobby actions, but do not isolate hostile network
traffic between VPN members. Do not operate this design as an open public or
multi-tenant service. Supporting unknown users safely requires separate network
segments and a different threat model.
The Companion lobby lock is coordination metadata, not a network firewall; it
does not block direct access to a host's game port from the shared VPN.

# Politique de sécurité — Français

## Signaler une vulnérabilité

Signalez les vulnérabilités en privé avec la fonction GitHub **Security >
Report a vulnerability** lorsqu'elle est disponible. Ne publiez jamais dans
une issue des credentials, des clés privées, des profils VPN, des adresses IP
publiques, des bearer tokens ou des identifiants de diagnostic TPM/CNG.

## Fichiers qui ne doivent jamais être commités

Ne publiez jamais les fichiers générés par un serveur déployé, notamment :

- `/root/openvpn-pki/` ;
- `/var/lib/openvpn-lan-party/enrollment/` ;
- `/var/lib/openvpn/credential-registry.json` provenant d'un vrai déploiement ;
- chaque `companion.json` généré et son bearer token ;
- l'état de déploiement `/etc/openvpn-lan-companion/` ;
- les certificats clients, clés privées, profils `.ovpn` ou réponses
  d'enrôlement.

Le dépôt contient uniquement des templates. Chaque déploiement doit générer une
nouvelle autorité de certification et des identités de joueurs uniques.

## Frontière des membres de confiance

Remettez un profil VPN uniquement à une personne connue et fiable pour
l'administrateur. Tous les joueurs connectés partagent le même LAN virtuel de
niveau 2. Les droits du Companion protègent l'identité et les actions de salon,
mais n'isolent pas le trafic réseau d'un membre hostile. N'exploitez pas cette
architecture comme service public ou multi-tenant : accueillir des inconnus en
sécurité exige plusieurs réseaux cloisonnés et un autre modèle de menace.
Le verrou d'un salon Companion est une règle de coordination, pas un pare-feu :
il ne bloque pas l'accès direct au port de jeu de l'hôte depuis le VPN partagé.
